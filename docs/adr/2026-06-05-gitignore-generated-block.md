# 2026-06-05 — Generate the project `.gitignore` Trellis block instead of appending a template

## Context

Trellis seeds each registered project with machine-absolute inheritance symlinks
(`.claude/skills/*`, `.agents/skills/*`, rules, commands, …) whose targets live
under `$TRELLIS_ROOT`. Those targets differ per machine, so the symlinks must be
gitignored; everything else Trellis seeds (hooks, `settings.json`,
`primers/INDEX.md`, skill-local configs, the *relative* `AGENTS.md` → `CLAUDE.md`
symlink) is project-state and should be tracked.

The old mechanism (`ensure_gitignore_fragment`) cat-appended a static template,
`core-rules/templates/project.gitignore.fragment`, guarded by a version sentinel
embedded in the block header (`(7-skill set + presets …)`, `(10-skill set …)`,
`(11-skill set …)`). When the canonical skill set changed, the sentinel changed,
and the next re-onboard **appended a new block alongside the old ones** rather
than replacing them. Over a year this produced 1–4 **stacked, stale** Trellis
blocks per project (clusterbid-console had project-authored ignore rules
*interleaved between* two Trellis blocks; neev had orphaned `se-core.md` /
`security-gate` lines that had escaped block boundaries; tgsc/lume had a stray
`# Trellis builder-skill symlinks` comment with loose paths). The stale blocks
omitted the current symlinks, so `execute`, `brainstorming`, `orchestrate`, and
`debrief` showed up as untracked across all 8 registered projects — at risk of an
accidental commit of machine-absolute symlinks.

The operator ask: each project `.gitignore` should ignore **only** the
hardcoded-symlink paths; every other Trellis file can (and already does) go into
git.

## Decision

Replace `ensure_gitignore_fragment` with **`write_gitignore_block`**, a
generate-and-replace mechanism.

- **Generated from reality, not a template.** `seed_symlink` records every link
  it creates whose target is **absolute** (`case "$target" in /*)`) into
  `IGNORE_PATHS`. The relative `AGENTS.md → CLAUDE.md` symlink is therefore
  excluded and stays tracked. At the end of the run (after all skills, commands,
  and presets are seeded) the block is written from `IGNORE_PATHS` — so it lists
  exactly the machine-absolute symlinks this project/harness combination actually
  has, with no template to keep in sync and no antigravity-`workflows/*` lines on
  projects where antigravity is off.
- **Version-agnostic marker.** The block header drops the skill-count sentinel
  (`# --- Trellis inheritance symlinks (per-machine; regenerated by
  onboard-project.sh) ---`). The count was the original stacking trigger and is
  meaningless once the block is rewritten in full every run.
- **Self-healing strip.** Each run removes all prior Trellis-managed blocks before
  writing the fresh one. A block begins at any line matching the prefix
  `# --- Trellis inheritance symlinks` and ends at **either** `# --- end Trellis
  fragment ---` **or** the legacy `# --- end SE Core fragment ---` variant. The
  nested per-session-state sub-block is swallowed by staying in-block until the
  fragment end. An exact-match **orphan sweep** then drops any surviving
  Trellis-owned line (the canonical symlink paths + legacy `se-core.md` + the
  stray builder-skill comment) left stranded between old stacked blocks.
- **Per-block, not span-based.** Stripping removes each `begin → end` region
  individually, so project-authored content interleaved between stacked blocks
  (clusterbid-console) is preserved. The orphan sweep matches only exact strings
  no project would ever author, so it cannot touch project content.
- **`.claude/session-autonomy` stays ignored.** It is per-developer, per-session
  runtime state — not a symlink and not a shippable file — so it remains in the
  generated block's per-session-state sub-block.

The static template `core-rules/templates/project.gitignore.fragment` is deleted;
`engineering-process.md` and the `rollout-*` operator hints that told operators to
`cat` it are updated to "re-run `onboard-project.sh`".

## Consequences

- **The stacking drift class is closed**, not reset at "11." Re-onboarding any
  project now collapses its stacked/stale/orphaned blocks into one clean,
  current block. The fleet rollout was one PR per project (8 total), each a pure
  `.gitignore` change validated by a safety assertion (no non-Trellis bare-path
  line may be removed) — proven against all 8 base-branch `.gitignore`s before
  any mutation, then run via real onboard in throwaway worktrees so live/dirty
  feature-branch checkouts were never touched.
- **Dual-maintenance is gone.** There is no longer a template to bump in lockstep
  with the `seed_symlink` call list; the ignore list is derived from the calls.
- **Minor cosmetic churn**: consolidating to one block at the end can relocate
  project lines that previously sat *after* a Trellis block (clusterbid-console's
  `.claude/screenshots/` et al.). `.gitignore` is order-independent, so this is
  functionally inert.
- **Ordering change**: the block is written *after* symlink creation (it must read
  `IGNORE_PATHS`), where the old append ran first. Within a single run the end
  state is identical; only an interrupted run between symlink creation and block
  write would briefly leave symlinks un-ignored.

## Status

Accepted — 2026-06-05. Mechanism shipped on `chore/gitignore-managed-block`;
rolled out to all 8 registered projects (neev, tgsc, akaushik.org, curat.money,
vericite, lume, clusterbid-console, gwtf-in), one PR each.
