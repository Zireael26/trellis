# 2026-05-29 — Opus 4.8 prompting best-practices: incorporation

## Context

Anthropic published a consolidated [Prompting best practices](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices) guide alongside the Opus 4.8 release (one reference covering Opus 4.8 / 4.7 / 4.6, Sonnet 4.6, Haiku 4.5). Opus 4.8 is the Trellis daily driver, so the guide was reviewed for incorporation.

A gap analysis found the guide largely **validates** Trellis rather than contradicting it. The autonomy slider, the bright-line guardrails, the context-log / primer state system, the surgical-scope code-quality rules, and the terse-communication rules already implement most of the guide's system-prompt patterns as infrastructure. Three classes of change were worth adopting; the rest is either already-done or API-only.

## Decision

Bundle the incorporation as one PR (precedent: `docs/adr/2026-05-18-anthropic-best-practices-bundled-rollout.md`), bump `core-rules/VERSION` 0.5.0 → 0.6.5, and mirror to the public template after the instance PR is green.

## What was incorporated

1. **Effort default → `xhigh` (the one real config change).** `core-rules/templates/claude-settings.json` now sets `"effortLevel": "xhigh"`. Verified against the Claude Code [model-config docs](https://code.claude.com/docs/en/model-config): `effortLevel` is a real settings field accepting `low|medium|high|xhigh` (`max` and `ultracode` are session-only and rejected there). Opus 4.8's Claude Code default is `high`; the guide recommends `xhigh` for coding/agentic work. It falls back to `high` on models that don't support `xhigh` (Sonnet 4.6, Opus 4.6), so the template is fleet-safe.

2. **Rule calibration in `CLAUDE.md` (surgical — three deltas).** (a) no-speculative-defensive-code / validate-only-at-boundaries (Code quality); (b) investigate-before-answering / never-claim-about-code-you-haven't-opened (Debugging); (c) a dispatch note — Opus 4.8 *under*-dispatches subagents and tools by default (the reverse of 4.6's over-spawning), so the existing dispatch triggers now counteract the model rather than reining it in, plus batch independent tool calls. Everything else the guide recommends as a parent-rule line was already present; no churn.

3. **New `docs/opus-4.8-steering.md`.** The verbose, reusable snippet library plus effort / thinking / model-identity guidance, each mapped to the Trellis surface that already implements it. The verbose snippets live here, not in the always-injected parent `CLAUDE.md`.

4. **Doc-only cross-references.** `autonomy.md` gains an "Opus 4.8 alignment" note (the L1–L5 slider *is* the guide's default-to-action ↔ conservative-action spectrum; the guardrails *are* its balancing-autonomy-and-safety advice). `engineering-process.md` gains: §8.6 don't-hard-code-to-tests; §14.3 "write rules for the current model" (4.8 literalism → state scope explicitly, positive > negative examples, sparing `CRITICAL:`/`MUST`); §5.2 a code-review coverage-not-filtering note. The `code-review-subagent.sh` header contract documents the coverage-not-filtering reviewer prompt.

## What was deliberately NOT incorporated

- **API-only mechanics.** Prefill-response migration, sampling parameters, and the `thinking: {type:"adaptive"}` / `budget_tokens` / `output_config` configuration are Messages-API concerns. Trellis steers Claude Code, where effort is the `effortLevel` setting and thinking is adaptive automatically — none are actionable here.
- **`alwaysThinkingEnabled: true`.** The guide warns that large or complex system prompts over-trigger thinking; Trellis injects a large prompt (parent rules + skills + hooks + primers), so forcing thinking on fleet-wide is the wrong default. Effort is the lever instead.
- **The long pre-4.8 frontend snippet.** The guide itself deprecates it for 4.8. Only the short `<frontend_aesthetics>` snippet is captured, in the steering doc (frontend-design is not a Trellis-core skill).
- **Parent-`CLAUDE.md` bloat.** "Comprehensive" was scoped to *cover every mappable concept* (in the steering doc), not *add a parent-rule line per doc section*. Each `CLAUDE.md` edit had to change behavior beyond text already present, or it became a steering-doc map entry instead.

## Framing note (honesty)

The code-review change is **documentation**, not a wired behavior fix. `code-review-subagent.sh` is an unwired skeleton — dispatch fires only if a project ships an executable `.claude/agents/code-reviewer.sh`. The hook's two-tier design (`critical` blocks, the rest is advisory) already *is* the guide's separate-filter-stage; the change instructs whoever writes the project-local reviewer to prompt for coverage. The optional `confidence` field is documented but not consumed by the hook today.

## Carve-outs

- **VERSION collision with `feat/antigravity-third-harness` — resolved.** Both branched from `main@0.5.0` and bumped to 0.6.0. AntiGravity (PR #71) merged first and kept `v0.6.0`; per the carve-out this PR rebased onto main and re-versioned to **`v0.6.5`**, resolving the `VERSION` / `CHANGELOG` conflict (antigravity's 0.6.0 entry retained; the Opus 4.8 entry sits above it as 0.6.5).
- **PR size.** Well under the 800-line hard cap; no size carve-out needed.

## Consequences

- Reversible at the commit boundary — each phase is a self-contained commit, revertable via `git revert <sha>`.
- The effort default reaches a project only when it next scaffolds or re-syncs its `.claude/settings.json` from the template; existing projects are unaffected until their normal upgrade path runs.
- Audit posture unchanged. The steering doc is referenced from `engineering-process.md` References and the `code-review-subagent.sh` header, keeping the rules ↔ docs invariant intact.
