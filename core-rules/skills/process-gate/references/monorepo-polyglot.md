# Reference — `monorepo-polyglot` stack profile

Long-form reference for the `monorepo-polyglot` profile sketched in `references/stack-profiles.md`. Worked example throughout: `clusterbid-console`.

Authoritative cross-cutting source: `engineering-process.md` §14 (Rule of Three) and §6 (Git workflow).

## 1. When to pick `monorepo-polyglot`

Pick this profile when the domain forces more than one production language to ship inside a single repo. The trigger is a hard ecosystem constraint, not a code-sharing preference.

- Pick `monorepo-polyglot` when:
  - A language is unavoidable at a system boundary (e.g., Python at the GPU boundary because vLLM / SGLang have no Go equivalent).
  - At least two of those languages own production services, not just dev tooling.
  - Cross-language contracts (proto, OpenAPI) need a single source of truth.
- Stay with the single-language profile (`monorepo-pnpm`, `monorepo-go`, `service-node`, etc.) when:
  - One language could carry the load and the second is only build glue or scripts.
  - The "second language" is a one-off CLI generated by `go:embed` or `node --experimental-strip-types`.

Decision examples:

| Shape | Profile |
|---|---|
| Go services + Python ML runners + Next.js frontend | `monorepo-polyglot` |
| Node API + Python scripts in `scripts/` | `monorepo-pnpm` (or `service-node`) |
| Go workspace, no other languages | `monorepo-go` |
| Single Python service, single language | `service-python` |

If the choice is borderline, default to the single-language profile and promote to polyglot only when the second language gains its first production service.

## 2. Subtree layout

Strict per-language root directories. Each language gets exactly one home; mixing is rejected by the orchestrator-coverage check.

clusterbid layout:

```
clusterbid-console/
├── services/                  # one directory per service; service.yaml declares language
│   ├── auth-service/          # Go
│   ├── gateway/               # Go
│   ├── inference-runner/      # Python (vLLM/SGLang) wrapped in a Go sidecar
│   ├── metering/              # Go
│   ├── model-catalog/         # Go
│   └── router/                # Go
├── pkg/                       # Go shared libraries (captoken, httpx, logger)
├── ts/                        # TS shared libraries (empty today; slice 5)
├── py/                        # Python shared libraries (empty today)
├── clients/                   # frontend apps + generated SDKs (pnpm workspace)
├── proto/                     # cross-language contract source of truth
├── infra/                     # IaC (Pulumi/Terraform), per-env
├── charts/                    # Helm charts
├── tools/                     # repo-lint, seed scripts, etc.
└── docs/                      # ADRs, engineering standards, RFCs, architecture
```

Hard rule: new top-level directories require eng-lead approval. The root Makefile fans out by directory name; silent additions create orphan subtrees the orchestrator can't reach.

Per-service `service.yaml` (see §10) carries `language:` so language-aware tooling does not need to grep for `go.mod` / `pyproject.toml` / `package.json`.

## 3. Contract layer (`proto/`)

`proto/` is the single source of truth for cross-language interfaces. `buf generate` produces Go server stubs, TS clients, and a Python SDK from the same `.proto` files.

- Generated code is checked into git. CI is deterministic; downstream services build without running codegen.
- Staleness is gated by a project-local `check-proto-contract.sh` validator. It re-runs `buf generate` into a tempdir and diffs against the committed output; non-empty diff fails the gate.
- Hard rule: no service-to-service JSON contracts outside `proto/`. JSON shapes drift silently — a renamed field becomes a billing bug when the consumer keeps reading the old name and gets the zero value.
- `buf.yaml` lives at `proto/buf.yaml`; `buf.gen.yaml` at `proto/buf.gen.yaml`. Outputs land under `clients/sdk-ts/`, `pkg/proto/`, and `py/clusterbid_proto/` respectively.

Breaking-change discipline: `buf breaking --against '.git#branch=main'` runs in CI and blocks any field renumber, type narrowing, or removal without a major-version bump.

## 4. Root `Makefile` as orchestrator

One root `Makefile` is the single entry point per verb. Trellis process-gate calls the Makefile only — it never invokes `go`, `pnpm`, `uv`, or `pytest` directly. This keeps `local.config.sh` language-agnostic and absorbs future language additions without re-onboarding.

Canonical verbs:

```makefile
# Top-level fan-out (simplified)
vet:
	@for d in $(GO_MODULES); do (cd $$d && go vet ./...); done
	@pnpm -r --filter '...' typecheck
	@uv run mypy $(PY_TREES)

lint:
	@for d in $(GO_MODULES); do (cd $$d && golangci-lint run); done
	@pnpm -r --filter '...' lint
	@uv run ruff check $(PY_TREES)

test:
	@for d in $(GO_MODULES); do (cd $$d && go test ./...); done
	@pnpm -r --filter '...' test
	@uv run pytest $(PY_TREES)

build:
	@$(MAKE) -C services/gateway build
	@$(MAKE) -C services/inference-runner build
	@pnpm -r --filter '...' build

dev:
	@./tools/dev/up.sh
```

Per-service Makefiles handle service-level builds (container, binary, image push). The root Makefile composes them; it does not duplicate logic.

Why the loop per Go module: `go vet ./...` from the workspace root is not supported — workspaces forbid running build commands at the workspace root, you must `cd` into each module. The orchestrator owns that loop so callers never know about the constraint.

`local.config.sh` for a polyglot project:

```bash
PROCESS_GATE_STACK_PROFILE="monorepo-polyglot"
PROCESS_GATE_TYPECHECK_CMD="make vet"
PROCESS_GATE_LINT_CMD="make lint"
PROCESS_GATE_TEST_CMD="make test"
```

## 5. Per-language toolchain

### 5.1 Go

- `go.work` at repo root listing every workspace module. Each module has its own `go.mod` with `replace` directives pointing at the local `pkg/*` paths (workspace mode resolves them locally, but the replace block keeps non-workspace builds reproducible).
- Go 1.22 minimum.
- `golangci-lint` at `.golangci.yml`. clusterbid's enabled linters: `errcheck`, `revive`, `gosec`, `staticcheck`, `gocritic`, `unparam`, `govet`, `ineffassign`, `gofmt`, `goimports`.
- Per-module `go test ./...`; never `go test ./...` at workspace root (same workspace constraint as `go vet`).

```
// go.work
go 1.22

use (
    ./pkg/captoken
    ./pkg/httpx
    ./pkg/logger
    ./services/auth-service
    ./services/gateway
    ./services/inference-runner
    ./services/metering
    ./services/model-catalog
    ./services/router
    ./tools/repo-lint
)
```

### 5.2 TS / Next.js

- `pnpm-workspace.yaml` declares every TS package. clusterbid: `clients/*` and `ts/*`.
- Turborepo (`turbo.json`) for task caching across the workspace. Pipeline stanzas wire `typecheck`, `lint`, `build`, `test`.
- `tsc --noEmit` for typecheck. ESLint or Biome for lint — pick one and stick with it; mixing produces conflicting autoformat behavior.
- Every app and every shared lib gets exactly one entry in `pnpm-workspace.yaml`. Nested workspaces are forbidden.

```yaml
# pnpm-workspace.yaml
packages:
  - "clients/*"
  - "ts/*"
```

### 5.3 Python

- One root `pyproject.toml` carrying shared `[tool.ruff]`, `[tool.mypy]` (with `strict = true`), and `[tool.pytest.ini_options]`. Per-service `pyproject.toml` overrides where the service needs different deps or Python version.
- Lockfile: `uv.lock` preferred (fast, deterministic, single tool for env + deps); `poetry.lock` accepted for legacy services.
- Python 3.12 minimum.
- Mypy `--strict` is non-negotiable per the project's coding standards. New code without type annotations fails typecheck.

```toml
# pyproject.toml (root)
[tool.ruff]
target-version = "py312"
line-length = 100

[tool.mypy]
strict = true
python_version = "3.12"

[tool.pytest.ini_options]
testpaths = ["py", "services/inference-runner", "services/fine-tuning-orchestrator"]
```

## 6. CI: per-subtree triggers

Every language subtree gets its own GitHub Actions workflow with a `paths:` filter scoped to its directory plus cross-cutting deps (`proto/`, root configs). Cross-language jobs live in a separate workflow that always runs.

Pattern:

- One workflow per service for service-level build / push / deploy: `svc-gateway.yml`, `svc-inference-runner.yml`, etc. `paths:` includes `services/<svc>/**`, `pkg/**` (for Go), `proto/**`, root configs.
- One monorepo-wide workflow for cross-cutting lint / format / orchestrator coverage / proto-staleness: `monorepo-ci.yml`. Always runs on every PR.

```yaml
# .github/workflows/svc-gateway.yml (excerpt)
on:
  pull_request:
    paths:
      - "services/gateway/**"
      - "pkg/**"
      - "proto/**"
      - "go.work"
      - ".github/workflows/svc-gateway.yml"
```

clusterbid's `monorepo-ci.yml` + `svc-gateway.yml` is the model. New services start by copying `svc-gateway.yml` and adjusting paths.

A project-local `check-ci-subtree-routing.sh` validator (see `stack-profiles.md`) verifies every language subtree has a matching `paths:` entry and flags cross-language workflow leakage (e.g., a Go service workflow triggering on `clients/**`).

## 7. Native git hooks

Use native git hooks (not husky) for polyglot projects — husky's pnpm-centric assumption fights the orchestrator pattern. Bootstrap by copying `core-rules/husky/{commit-msg,pre-push}` and replacing the pnpm tail with `make vet && make lint`.

Wire-up:

```bash
git config core.hooksPath .githooks
chmod +x .githooks/*
```

Scripts:

- **`commit-msg`** — conventional-commits validator. Scope must match one of: a pnpm workspace package name, a Go module name, a Python package name, or a service directory name. Per-project allowlist lives in `.githooks/scopes.txt`.
- **`pre-push`** — Trellis PR-flow guard first (block direct push to `main`/`master`; allow `TRELLIS_ALLOW_MAIN_PUSH=1` override per `engineering-process.md` §6), then `make vet && make lint`. The Makefile owns per-language fan-out; the hook stays language-agnostic.
- **`pre-commit`** — deferred. Polyglot-specific pre-commit validators (proto-staleness, generated-code drift, service.yaml lint) are still in design. Add them only once they hit Rule of Three across at least three polyglot projects.

Pattern reference: Lume's `.githooks/` directory shows the layout, exit-code conventions, and the `TRELLIS_ALLOW_MAIN_PUSH` escape hatch.

## 8. PR size budget

Polyglot PRs trend large because cross-language refactors cross boundaries: a single schema change touches the Go server stub, the TS client, the Python SDK, and at least one consumer in each. Expect to bump the limits over time as the codebase grows.

Starting defaults (clusterbid's current setting):

```bash
PROCESS_GATE_PR_SIZE_LIMIT=400   # soft
PROCESS_GATE_PR_SIZE_HARD=800    # hard
```

Generated code under `pkg/proto/`, `clients/sdk-ts/`, and `py/clusterbid_proto/` should be marked `linguist-generated` in `.gitattributes` so it does not count against the budget. Without that mark, every proto change blows the soft limit.

When a cross-language refactor legitimately exceeds the hard cap, land an ADR under `docs/adr/` naming the change and explaining why splitting would make review or rollback less clear. The gate accepts the override only when the diff includes a new ADR.

## 9. ADR location

ADRs live at `docs/adr/`, numbered `NNNN-<slug>.md`. Cross-language decisions always get an ADR:

- "buf for codegen" — why buf over protoc, raw generators, or hand-written stubs.
- "Python only at the GPU boundary" — the constraint that gates new Python services.
- "Turborepo vs Nx" — pick-one rationale for the TS task runner.
- "uv vs poetry" — Python lock-file and env-manager choice.

clusterbid's `docs/adr/0001-slice-1-bundle.md` and `docs/adr/0002-slice-2-bundle.md` are the existing pattern. Each ADR has: Context, Decision, Consequences, Alternatives considered.

Set in `local.config.sh` so the PR-size hard-cap override knows where to look:

```bash
PROCESS_GATE_ADR_DIR="docs/adr"
```

## 10. Service manifest discipline

Every service has a `services/<svc>/service.yaml`. CI rejects services without it; the orchestrator coverage validator reads it to discover which subtrees need fan-out.

Minimum required fields:

```yaml
# services/gateway/service.yaml
name: gateway
language: go
owner: platform
slo:
  availability: 99.9
  latency_p99_ms: 250
public: true
scope: gateway
depends_on:
  - auth-service
  - router
db_schemas:
  - gateway
```

Field meanings:

- `name` — canonical service name; must match the directory name.
- `language` — `go` | `python` | `typescript` | `rust`. Drives orchestrator fan-out and CI workflow selection.
- `owner` — team or individual responsible.
- `slo` — at minimum availability + one latency or throughput target. Used by the operations runbooks.
- `public` — true if the service is reachable from outside the cluster.
- `scope` — Conventional-Commit scope name; commit-msg hook validates against this.
- `depends_on` — list of upstream services this one calls. Used by the dependency-graph drift check.
- `db_schemas` — list of schemas this service owns. Used by the migration-safety validator.

Pattern source: `docs/engineering/repo-structure.md` in clusterbid-console. Extend the schema in that doc, not per-service.

## 11. Promotion path

`monorepo-polyglot` is currently n=1 — Lume-style carve-out, clusterbid-console is the sole adopter. Per `engineering-process.md` §14.1 (Rule of Three), all polyglot-specific validators stay project-local under `<project>/.claude/skills/process-gate-local/scripts/` and `<project>/.agents/skills/process-gate-local/scripts/` until a second polyglot project lands.

When the second polyglot project onboards:

1. Diff the validators between the two projects. Anything byte-identical or trivially parameterizable is a promotion candidate.
2. Wait for a third witness before lifting. n=2 is a coincidence; n=3 is a pattern.
3. On n=3, lift per `engineering-process.md` §14:
   - Move the validator into `$TRELLIS_ROOT/core-rules/skills/process-gate/scripts/`.
   - Add a corresponding reference under `references/`.
   - Update `references/stack-profiles.md`.
   - Run the extended `parent-hook-drift` audit to verify byte-identity across all three adopters.
4. Track candidate validators in `core-rules/deferred.md` so they are not forgotten between project onboardings.

Canonical six gates (PR hygiene, secrets, bypass, tests, docs, stack profile) apply unchanged in the meantime.
